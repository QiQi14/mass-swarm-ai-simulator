# Strategy Brief: Zero Observation Channels & ECP Calculation Failure

## Problem Statement
A fresh training run was initiated, but the system logged critical failures where `ch1`, `ch2`, and `ch3` (enemy count, friendly ECP, enemy ECP) were all zeros. Additionally, the user requested a fresh start (Stage 0) but the script was launched at Stage 1 due to outdated tracking in `TRAINING_STATUS.md`. We must diagnose the root cause of the observation pipeline failures and outline a path to restart correctly.

## Hypotheses & Analysis

### Hypothesis 1: Terrain Payload Mismatch (The `len(flat_data) != active_size` silent drop)
**Test:** Trace Python `ResetEnvironment` terrain generation -> ZMQ payload -> Rust initialization -> Python Vectorizer.
**Evidence:** 
- In `macro-brain/src/env/swarm_env.py`, `terrain` is generated by importing `generate_terrain_for_stage` from `src.utils.terrain_generator`. For Stages 1 & 2, this function returns `None` (for "flat terrain").
- Because Python sends `{"terrain": null}` in the `ResetEnvironment` payload, the Rust micro-core engine skips updating the `TerrainGrid` resource inside `micro-core/src/bridges/zmq_bridge/reset.rs` (it requires `Some(tp)`).
- Consequently, Rust's `TerrainGrid` remains at its default startup size (50x50, 2500 cells).
- Rust then packages density maps as 2500-length arrays and sends them back to Python.
- Python expects the density array to match `active_size` (which is 625 for Stage 1's 25x25 grid). In `src.utils.vectorizer.py`, `_place_density` checks `len(flat_data) != active_size` and simply `return`s silently, leaving the CNN channels as all zeros.
**Verdict:** Correct. This causes all density and ECP maps to be dropped by Python.

### Hypothesis 2: ECP Calculation Missing `ecp_stat_index` Payload
**Test:** Trace ECP calculation in Rust (`micro-core/src/systems/state_vectorizer.rs`) and how `ecp_stat_index` is configured from Python.
**Evidence:** 
- Rust computes ECP using `density_config.ecp_stat_index`. If this is `None`, it defaults to `0.0`, resulting in exactly 0.0 ECP for every entity.
- In `macro-brain/src/env/swarm_env.py`, the `payload` generated for `ResetEnvironment` simply omits the `"ecp_stat_index"` key entirely.
- Rust's `ResetRequest` structure defines `pub ecp_stat_index: Option<Option<usize>>`. Without it in the JSON payload, it defaults to `None`.
**Verdict:** Correct. This mathematically forces all ECP distributions to `0.0` before they are even sent to Python.

## Root Cause
1. **Grid size mismatch:** Python sends a `null` terrain payload for flat maps, causing Rust to use a 50x50 default map instead of dynamically resizing to the Stage map dimensions, causing Python to discard the 2500-cell array payload.
2. **Missing ECP initialization:** Python fails to inject the resolved `ecp_stat_index` into the ZMQ reset payload, causing Rust to multiply weapon damage by 0.0 HP.

## Recommendations

### Recommended Changes
1. **Fix Terrain Fallback Payload:** Modify `macro-brain/src/env/swarm_env.py` to import `generate_terrain_for_stage` from `src.training.curriculum` instead of `src.utils.terrain_generator`. The curriculum version correctly synthesizes a purely flat `TerrainPayload` dict properly sized to `active_grid_w` and `active_grid_h`.
2. **Inject `ecp_stat_index`:** Add `"ecp_stat_index": self._primary_stat_index` to the `payload` dictionary in `swarm_env.py` inside the `reset()` method.
3. **Train Fresh from Stage 0:** Re-run `train.sh` without the `--start-stage 1` parameter so the system naturally flows from Stage 0 without arbitrary assumptions.

## Brute-Force Analysis
Unrelated to curriculum level design; this was a critical data pipeline bug. Without data passing correctly, the agent would rely on pure random coin-flip behavior rendering any stage unsolvable.

## Impact on Later Stages
Fixing the dynamic resizing logic ensures that Stages 0-3 (which utilize smaller bounds like 20x20 and 30x30 to accelerate training) accurately map density information to the core tensor without array dimension mismatch errors.
